## A. 運用特化AI（統括部コア提案）

### A-1. 障害予見・対応の自動化

**課題**: 障害検知後の初動対応・原因特定に時間がかかる。過去の類似障害の知見が個人の頭の中に留まっている。

**AI活用イメージ**:

```
ログ・監視データ
    ↓
AI（過去の障害DBと照合）
    ↓
「3時間後に類似の障害が起きる可能性あり」  ← 予見アラート
「この障害パターンは過去17件。対応手順はXX」 ← 初動ガイダンス
「同種インシデントの平均対応時間: 42分」    ← KPI自動記録
```

**実装例（佐藤が関わっている案件）**:
- 保守運用現場にSlack連携のAI-PMエージェントを導入
- 毎日の障害状況を自動サマリー → 担当者へプッシュ通知
- 週次で「再発リスクの高いインシデントTop3」を自動レポート

**世界の導入実績**:

| 企業・機関 | 業界 | 効果 | ツール |
|-----------|------|------|--------|
| **ソフトバンク**（日本） | 通信 | 保守対応時間 **23分30秒 → 2分30秒**（89%削減） | 独自AI異常検知+根本原因分析 |
| **KDDI**（日本） | 通信 | 年間数千万円コスト削減、重大事故未然防止 | 時系列予測AI |
| **Microsoft Azure** | クラウド | 障害対応時間（TTM）**38%短縮**、AI精度90% | Azure AIOps Agent |
| **米国大手金融機関** | 金融 | MTTR **60%削減**、1件あたり$85,000の損失回避 | AIOps統合基盤 |
| **PagerDuty × AWS** | SaaS/クラウド | 対応時間 **45〜60分 → 2〜3分**（95%削減） | PagerDuty + Logz.io |
| マルチクラウド基盤 | インフラ | 稼働率99.99%達成、重大障害 **43%削減** | 統合CRP |

> **日本の通信大手（ソフトバンク・KDDI）が先行実践中。NECのインフラ・通信業種の顧客にとって説得力のある前例が揃っている。**

**期待効果**: 障害対応時間60〜90%短縮（国内通信キャリア実績）、ナイトシフト負荷軽減、1件あたり数百万円規模の損失回避

### A-2. 問い合わせ業務の自動化

**課題**: 同じ質問が繰り返し来る。担当者が回答を毎回手書きしている。FAQが陳腐化している。

**AI活用イメージ**:

```
問い合わせ（Teamsチャット / メール）
    ↓
AI（社内手順書・過去対応履歴を参照）
    ↓
一次回答を自動生成 → 担当者がレビュー＆送信
    ↓
解決率・未回答パターンを蓄積 → FAQ自動更新
```

**実現方法**: Copilot Studio × SharePoint（既存M365環境で可）

**世界の導入実績**:

|-----------|------|------|--------|
| **アサヒグループHD**（日本） | 食品飲料 | 社内問い合わせ **72,000件 → 36,000件以下**（50%削減） | 社内AIヘルプデスク |
| **三菱商事**（日本） | 総合商社 | 正答率 **67% → 89%**（+22pt）、オペレーター工数削減 | IBM Watson |
| **Databricks** | クラウドデータ | IT部門から人事・法務・L&Dなど **8部門へ横展開** | Freshservice AI |
| **Freddy AI（小売業）** | 小売 | 初回応答時間 **12分 → 12秒**（99%削減）、解決時間 1時間→2分 | Freshservice |
| **AssemblyAI** | B2B SaaS | 初回応答時間 **97%削減**、自動解決率60%向上 | AI-first support |
| Gartner調査（複数社） | 全業種 | ROI平均 **$3.50/投資$1**（トップ企業は8倍） | 各種AIプラットフォーム |

**期待効果**: 問い合わせ件数50%削減、対応時間60〜99%短縮、24時間一次対応の自動化

### A-3. 仕様書・手順書の作成・レビュー自動化

**課題**: 運用手順書の更新が追いつかない。担当者交代のたびにドキュメントが消滅する。レビューが属人的で品質にばらつきがある。

**AI活用イメージ**:

```
【作成】
既存の手順書 + ログ + 担当者へのインタビュー
    ↓ AI
ドラフト手順書を自動生成（Markdown / Word）
→ 担当者が確認・加筆

【レビュー】
新規作成の仕様書 / 手順書
    ↓ AI
「前回版との差分」「抜け落ちているステップ」「曖昧な表現」を指摘
→ 品質チェック工数を90%削減

【継承】
退職・異動前の担当者の知識を構造化インタビューでドキュメント化
→ ナレッジ断絶を防止
```

**世界の導入実績**:

| 企業・機関 | 業界 | 効果 | ツール |
|-----------|------|------|--------|
| **ミスミグループ** | 製造業（日本） | 技術仕様書作成コスト **50%削減**、標準化品質で属人性排除 | AI文書生成基盤 |
| **NTTドコモ** | 通信（日本） | 開発者の **78%が週1時間以上**削減、手順書・仕様書の更新サイクル短縮 | GitHub Copilot |
| **日立製作所** | 重電・インフラ（日本） | ドキュメント作成を含む開発生産性 **10〜20%向上** | GitHub Copilot |
| **都城市（宮崎県）** | 自治体 | 条例・規程文書作成のAI化で **年間1,800時間削減** | 行政AI文書化 |
| **金融大手（複数社）** | 金融 | コンプライアンス文書レビュー工数 **60〜80%削減** | AI Review Agent |
| McKinsey調査（複数社） | 全業種 | ナレッジワーカーのドキュメント作業時間 **最大60〜70%削減** | 各種LLM基盤 |

> **日本の大手（NTTドコモ・日立）が実績報告済み。NECの通信・インフラ領域とのシナジーが直接的。**

**期待効果**: ドキュメント作成工数50〜70%削減、レビュー工数90%削減、ナレッジ断絶リスク大幅低減

## B. AI駆動PM（2026年度AI推進PJの管理基盤）

### なぜ今、AI-PMなのか

2026年度、統括部では**各グループからAI推進要員を選出し、自部署の業務自動化PJを走らせる**体制になると想定されます。

このプロジェクトの管理自体をAIにやらせることで——

> **「AIで進めるプロジェクトを、AIで管理する」**

という体験を全参加メンバーが肌で感じられます。これが統括部全体への**AI浸透の最短ルート**です。

### AI-PMの具体的な動き

```
Slackチャンネル（各グループのPJチャンネル）
    ↓
AIエージェントが常時監視
    ↓
【朝】各自に「今日のタスクと優先度」をブリーフィング
【週次】「全体進捗 + ここに問題あり」を田岡さんにサマリー
【随時】タスク遅延・依存関係リスクを自動検知してアラート
【月次】「各グループの成果 + 次フェーズへの提案」をレポート
```

### 田岡さんのマネジメント工数

| 現状 | AI-PM導入後 |
|------|------------|
| 各チームを個別に状況確認 | AIサマリーを読むだけでOK |
| 進捗報告会議の準備に時間 | 自動生成レポートを確認 |
| 問題の把握が遅れがち | リアルタイムアラートで先手 |

お財布・社内承認の状況に応じて、複数の入り方を用意しています。

### ライト：定期相談コース

> **「まず話を聞いてほしい」「事例を教えてほしい」レベル**

- 月1回の定例会（1〜2時間）での相談・情報共有

**こんな方に**: 社内調整・稟議の準備フェーズ。費用・工数をかけずに情報収集したい。

### スタンダード：伴走コース

> **「プロトタイプを作って検証したい」「小さく始めたい」**

- 月2〜4回のミーティング + 非同期サポート

- 統括部AI推進メンバーへの技術移転・レクチャー

**こんな方に**: 上に成果を見せたい。手を動かしながら検証したい。

---

### フル：実装同行コース

> **「本格的にやる。実装まで一緒にやってほしい」**

- 週1以上の密な連携
- 複数ユースケースの設計〜実装〜運用設計まで一気通貫
- BAOOに加え、Unsonのエンジニアリソースを組み込み
