## ネタ1: 俺がClaude Codeに触れたのは3週間前 — Tasteは時間に比例しない

### 【哲学】

「俺はAI歴◯年だから」「Claude Code歴◯ヶ月だから」 — そういう発想が**時代遅れになった**。

AIで作れるハードルが消えた今、勝負は「**何を作るか決められるか**」だけ。それは Taste の深さで決まる。Taste は「好きで使った時間」に比例する（4/15）。**ツールに触れた時間じゃない**。

「Claude Code 使ってる人すごい」じゃなくて、「**何を作るか決めた人がたまたま Claude Code を使ってた**」が正しい。順番が逆。

### 【俺の経験】

4/20 のAI駆動経営イベント登壇で、俺はこう発言した記録が残ってる:

> 「Claude Codeの端末に入ったのは**3週間前ぐらい**」

つまり 4月頭〜中旬くらいから本格的に使い始めた。それから今日（5/7）まで 約1.5ヶ月。その間に何が起きたか:

- **BrainBase OSS化**: 1日50コミットで公開（参考: 2025-12-31 の準備、Claude Code触る前は別の手段）
- **MANA進化**: 議事録抽出/タスク分解/Graph SSOT統合
- **グローイン案件**: 月額アジャイル型に再設計
- **NEC研修案件**: AI推進者11名教育プログラム（5月末〜実施）
- **Story Driven Development を提唱**: 4/15登壇

1.5ヶ月でこの量。**理由は俺がClaude Codeに詳しいからじゃない**。「何を作るか」がもう自分の中で言語化されていたから、Claude Codeに渡したら3日で動くものになっただけ。

### 【あなたができること】

「ツール覚えてから何作るか考えよう」を**逆にしてください**。

今夜、紙とペンで5分:
1. 自分が一番「これは絶対に世の中に出したい」と思うものを1つ書く
2. なぜそれを出したいかの理由を3行で書く
3. それだけ書いて、5/12のセミナー受ける

これで Claude Code は最強のパートナーになる。**逆に、これがない状態でClaude Codeを覚えても、「すごいツール覚えたけど何も作れない人」になる**。

## ネタ2: 「AIで作れない」を前提にしてた時代の終わり

### 【哲学】

ほんの数年前まで、こうだった:
> 「これ作りたいけど、エンジニアいないから無理だな」  
> 「自分でコード書けないから諦めるか」

この **「作れない」を前提にした思考の癖**が、まだ多くの人に残ってる。

でも今は違う。「**作れない」前提を捨てて、『何を作りたいか』だけが問い**になる時代が来た。

これは Taste（4/15）の話に直結する。「作れない前提」だと、Taste がそもそも育たない。「**作れる前提**」になって初めて、「じゃあ何を作るのか？」という問いが立ち上がる。Taste はその問いを浴び続けた人にだけ育つ。

### 【俺の経験】

俺がここ1.5ヶ月で気づいたのは、**「作れない理由」を探す癖が一番のボトルネック**だったということ。

具体例:
- BrainBaseをOSS化した時、「セキュリティチェックの仕組みがないから無理」と思ってた
- でも Claude Code に「機密情報自動チェックシステム作って」と渡したら、その日に動いた（コミット `5b906c0e`）
- 「無理」じゃなくて「**作る前から諦めてた**」だけだった

別の例。MANAのn8n廃止（2025-12-29、7ヶ月運用してたツールを1日で捨てた）。  
Claude Code の前は「n8nから移行するのは大変そう」と思ってた。でも実際 Claude Code に「n8n依存を全部剥がして直接コミットに移行しろ」と渡したら、1日で完了した。

**「作れない」じゃなく、「作りたい」が言語化できてないだけだった**。これに気づいたのが俺の中での革命。

### 【あなたができること】

今あなたが「諦めてること」「作れないと思ってること」を**1つだけ思い出してください**。

例:
- 「自分用の家計管理アプリ作れたらな」
- 「うちの会社の議事録要約Bot欲しいな」
- 「子どもの英語学習に専用ツール欲しいな」

それを5/12のセミナーで Claude Code に渡してみてください。**3日後には動いてる可能性が95%**。  
セミナー後の懇親会で「実際に動いた」って報告を聞くのが俺の楽しみ。

## ネタ3: Taste を実装に落とす技術 — Story Driven Development

### 【哲学】

ただし、Claude Code に「ぼんやりしたお願い」を投げると、**ぼんやりしたものが返ってくる**。

俺が4/15のLT登壇で話した **Story Driven Development** が、ここで効く。

```
仕様駆動  → AIに「指示」を渡す → 認知負荷高い → 間違える
ストーリー駆動 → AIに「物語」を渡す → 認知負荷低い → 続きを想像できる
```

**「仕様」と「物語」の違い**:
- 仕様: 「ボタンをクリックしたらAPIを呼ぶ」
- 物語: 「俺は税理士の仕事を年100時間減らしたい。だから決算書から税額を3秒で出すAIが欲しい」

物語があると、Claude Codeは「**続きを想像**」できる。Tasteを物語の中に埋め込むと、AIはあなたのTasteで動くようになる。

### 【俺の経験】

俺が Claude Code に渡してる「物語」のサンプル（実際に俺の workspace にある）:

- `IDENTITY.md` — 俺は何者か（Judgment Core）
- `CLAUDE.md` — このプロジェクトはどういう前提で動くか（Frame）
- `_codex/` — 全プロジェクトの共有ナレッジ（Story 集）
- `decisions/` — 過去の意思決定ログ（Event ログ）

これらを Claude Code が読み込んでから動くから、**俺の判断基準で動く**。属人性を消したんじゃなくて、**属人性を Markdown にして AI に挿入してる**。

具体例:
- 「ライブのネタ帳を作って」だけだと、ぼんやりしたものが返ってくる
- でも `IDENTITY.md`（佐藤の世界観）+ `LIVE_2026-04-29.md`（前回ネタ帳）+ git履歴（実証エピソード）を渡すと、**「佐藤さんが書きそうなネタ帳」**が返ってくる
- 今日のこのネタ帳がまさにそれ（3層構造、git履歴ベース、Taste帰着）

### 【あなたができること】

5/12のセミナーまでに、**1ファイルだけ書いてください**。  
ファイル名: `me.md`

中身は:
```
# 俺の世界観（Frame）
- どんな世界に生きたいか
- 何を絶対しないか

# 今やってること（Story）
- 今週やってること
- 今月やってること
- 今年やりたいこと

# 直近の出来事（Event）
- 昨日何をしたか
- 何に困ってるか
```

書いたファイルを Claude Code に「これを前提に動いて」と渡せば、**Claude Code が「あなた」になる**。これが Story Driven の最小実装。

## ネタ4: AIで作れる時代の地雷 — 「自動化しすぎ」が殺す

### 【哲学】

「Claude Code で全部自動化しよう」は **死亡フラグ**。

俺は4月のSNS自動化プロジェクトの会議で、こう言った:
> 「自動化を進めすぎた結果、一旦お休みしようと思ってる。**ちょっと自動化しすぎるのも良くない**」

100%自動化を目指すと:
- アウトプットがあなたっぽくなくなる
- 「あれ、これ俺が判断したっけ？」が増える
- Tasteが入ってない出力が量産される
- 結局、最後に全部手直しすることになる

**Tasteを挿入する場所を残す**。これが Claude Code 時代の経営判断。

### 【俺の経験】

俺が Claude Code に**任せていない**領域:

- ❌ プロジェクトの戦略決定（誰と組むか、何を捨てるか）
- ❌ 顧客との初回ミーティング（信頼の起点は人）
- ❌ 値付け（5組限定の理屈は俺の哲学）
- ❌ 採用の最終判断
- ❌ ライブで何を話すか（ネタ帳の素材は集めてもらうが、芯は俺が決める）

逆に**任せてる**領域:
- ✅ 議事録抽出 → タスク化（MANA）
- ✅ コード実装（仕様が決まれば）
- ✅ ドキュメント整備（Story Driven）
- ✅ コミットメッセージ
- ✅ デバッグ・テスト
- ✅ リサーチ・要約

**16プロジェクト × 月5時間**を回せてるのは、この線引きを**頭で決めてるから**。Claude Code が決めてるんじゃない。

### 【あなたができること】

5/12のセミナー前に、**3つだけ書いてください**:

「**俺が絶対自分でやる3つ**」

例:
- 顧客との重要な意思決定の場
- 値段を決める瞬間
- ◯◯について発信する内容の最終判断

これを書いたら、それ以外は全部 Claude Code に任せていい候補。**Tasteの民主化はできない**。だからこそ、Tasteを置く場所を意識的に残す。

## ネタ5: Claude Code × OpenRouter — 「判断」と「実装」の分離

### 【哲学】

経営者にとっての一番の制約は **「PCの前にいる時間」**。移動中、会食中、家族との時間中に「判断はできるけど実装はできない」のがリアル。

Claude Code は PC が必要。でも判断は手のひらでやれる方がいい。

俺が4/20の登壇で示した使い分け:
- **OpenRouter（スマホ）** = 移動中の判断・思考整理・Slack返信
- **Claude Code（PC）** = 腰を据えた実装・複雑な構造変更

**「判断するときの俺」と「実装するときの俺」を別の道具で動かす**。これが時間の倍数化を生む。

### 【俺の経験】

具体的なフロー:
1. 移動中に Slack でMANAから「議事録抽出済み、タスク3件確認お願いします」が来る
2. スマホの OpenRouter で読んで「このタスクは却下、これは優先度高、これはBrainBaseに統合」と返事
3. 帰宅後、Claude Code を開いて「BrainBaseに統合するタスク、設計書いて」と渡す
4. 30分で動くプロトタイプができてる

これで**1日の有効稼働が3倍**くらいになった。「夜20時以降働かない」（4/22宣言）が成り立ってるのは、移動時間が判断時間として機能してるから。

### 【あなたができること】

5/12のセミナーは Claude Code が中心だけど、**OpenRouter（スマホ）も合わせて使う**ことを覚えて帰ってください。

簡単な始め方:
1. スマホに OpenRouter のアプリを入れる
2. Claude を選ぶ
3. 移動中に「今日の予定見直して、優先度TOP3を出して」と聞く
4. PCに戻ったら、その判断をベースに Claude Code で実装する

これで「判断と実装の分離」が成立する。**PCに張り付かなくていい人生**が手に入る。

## ネタ6: 5/12セミナーで何を伝えるか — 「使い方」じゃなく「渡し方」

### 【哲学】

普通の Claude Code セミナーは「**ツールの使い方**」を教える。  
俺の5/12セミナーは違う。「**Claude Code に何を渡すか**」を教える。

なぜなら、ツールの使い方は YouTube に山ほどある。でも「**自分の Taste を Claude Code に渡す方法**」は、誰も教えてくれない。

これがセミナーの差別化。  
「使い方」ではなく「**渡し方**」。Story Driven Development の実装版。

### 【俺の経験】

NEC の AI推進者教育（4/27議事録）でも、俺はこの構造で設計した:

```
1. 知る：外部事例で危機感を醸成
2. 選ぶ：業務マップにAI活用候補を貼り付け
3. 理解する：MCP・エージェント・RAG等の活用パターンと勘所
4. 作る：POCテーマ作成
```

**「ツールの使い方」は3番目のごく一部**。残りは全部「何を作るか」「どう渡すか」の話。

これを個人向けに圧縮したのが5/12セミナー。

### 【あなたができること】

5/12（来週月曜）に Zoom で会いましょう。  
**Claude Code 触ったことない人**でも大丈夫。むしろ、まっさらな状態で来てくれた方が、「渡し方」の本質が伝わる。

セミナー後、参加者には:
