## ネタ1: 仕組み化は「ツール追加」じゃない、「OSの入れ替え」だ

### 【哲学】

Windows 95にアプリをいくらインストールしても、速くならない。Windows 11にOSごと入れ替えるから速くなる。仕組み化も同じだ。

世の中の人は「Notion入れた」「ChatGPT契約した」「Zapier繋いだ」を仕組み化だと思ってる。違う。それは**Windows 95にアプリを足してる**だけ。

仕組み化の本質は「**意思決定プロセスそのものを再設計する**」こと。判断基準・情報構造・ナレッジを一体のOSとして入れ替える。だから時間がかかる。だから多くの人ができない。

### 【俺の経験】

MANA（俺のAIマネージャ）の最初のコミットを git で見ると、こう書いてある。

```
2025-05-19  fe15869  feat: bootstrap Bolt app
```

最初は **Slack分類ボット**だった。bug / feature / question にラベル付けるだけのやつ。当時の俺は「ツールを足した」つもりだった。

そこから9ヶ月。
- 2025-06-28 `192bd5b` 重複排除（最初の「型」）
- 2025-12-29 `a863917` n8n完全廃止（捨てる決断）
- 2026-01-22 `3b07597` M1〜M12型化（業務全体の型化）
- 2026-02-07 `51f533b` Graph SSOT統合（OSとしての完成）

途中で気づいた。**俺がやってたのはツール追加じゃなくて、自分の判断OSを丸ごと入れ替えてた**んだと。今のMANAは、俺の意思決定の型がそのままコード化されたもの。

### 【あなたができること】

今夜、1ファイルだけ書いてください。ファイル名は `my_os.md`。中身は3行でいい。

```
1. 俺はどんな判断基準で生きてるか（Frame）
2. 今やってる仕事は何か（Story）
3. 昨日何をしたか（Event）
```

これがあなたの「OS」のv0.1。仕組み化はここから始まる。Notionでも紙でもいい。**書き出した瞬間に、あなたは「OS入れ替え組」になる**。

## ネタ2: AIに仕様を渡すな、物語を渡せ

### 【哲学】

これは俺が4/15のLT登壇で話したテーマ。**Story Driven Development**。

仕様駆動 = 認知負荷が高い。バラバラな指示がたくさん。AIも人もミスする。  
ストーリー駆動 = 認知負荷が低い。物語があると「続き」を想像できる。本を半分読んだら続きが想像できるのと同じ。

仕組み化も同じだ。仕組みを「指示の集合」と捉える人は永遠に作れない。  
仕組みを「**物語の圧縮**」と捉えた瞬間に作れるようになる。

```
Frame（世界観・前提）  ← めったに変わらない
  ↓
Story（やること）     ← horizonごとに粒度が違う
  ↓
Event（起きたこと）   ← AIがオートで処理
```

**Frameを飛ばしてStoryを作らない。Storyを飛ばしてEventを実行しない。** この順番を守れる人だけが、仕組みを持つ。

### 【俺の経験】

BrainBaseをOSS化した時の git 履歴がリアルだ。

```
2025-12-31  bd27aacc  feat(oss): initial commit for Brainbase OSS
2025-12-31  d04c34d8  feat: OSS公開のための7つのBLOCKER修正
2025-12-31  5b906c0e  chore: OSS公開前の機密情報自動チェックシステム構築
2025-12-31  7e66b1f6  fix: 他のメンバー名（川合、渡邉、星野）のセキュリティチェック追加
... (1日で50コミット)
```

俺はOSS化を「機能の集合」として作ってない。**「公開する」というFrame**から始めた。
- Frame: 「これは公開して、コミュニティで育つもの」
- Story: 「だから機密情報が混ざってちゃダメ」
- Event: 「自動チェックシステムを作る」

順番が逆だったら、機密混入しながらコミットして詰む。Frameが先にあったから、1日でOSになった。

### 【あなたができること】

自分の仕事を **1段落の物語**にしてください。140字でいい。

例（俺の場合）:
> 「俺は AI ファースト経営の実例を作っている。MANAで自分のPM業務を75%削減し、BrainBaseをOSSとして公開しながら、その運用ノウハウを5組限定で売っている。仕組みを実証し、それ自体を商品にしている。」

書けない人は、まだFrameが定まってない。仕組み化以前の問題。書けたら、その物語を AIに渡してください。AIは「続き」を想像できる。あなたの仕事の自動化案を勝手に出してくる。**それが Story Driven 化の最初の一歩**。

## ネタ3: 力技で整える勇気を持つ — 民主的に進めると死ぬ

### 【哲学】

これは俺が3/24の登壇でも何度も話した。

> 「データを集めるところとか権限を振るところは、**力技でやるしかない**。1回整えてしまうと、あとはかなり綺麗に進む」  
> 「『皆で決めよう』じゃなくて、**トップがこれでやれって言っちゃう**のが一番いい」

仕組みを持てない人の8割は、ここで詰まる。「みんなの意見を聞いて決めましょう」と言って、何も決まらない。

仕組み化は**初期段階で集中負荷を投下する勇気**が必要。最初に1人が「これでいく」と決め切る。後から調整できる。でも最初に決めなかった仕組みは絶対に動き出さない。

### 【俺の経験】

BrainBase OSS化の2025-12-31、俺は1日で50コミットした。
- 機密情報削除
- セキュリティ自動チェックシステム構築
- メンバー名のチェック追加（川合、渡邉、星野）
- README全面刷新
- スクリーンショット追加

これ、誰かと相談してたら100年経っても終わらない。**俺が「今日OSにする」って決めて、1日で力技でやった**。だから動き出した。

MANAの n8n 廃止も同じ。2025-12-29 に1日で `a863917` `815c421` を含む7コミットで n8n を完全廃止した。「皆と相談」してたら7ヶ月の運用ツールは捨てられない。

### 【あなたができること】

来週、**1日だけ「仕組みデー」を取ってください**。半日でもいい。
- カレンダーに「仕組みデー」と書く
- その日は他の予定を全部断る
- 1日かけて「最初の構造」を1個だけ完成させる

ポイントは**他人に相談しない**こと。「これでいいか確認」もしない。**1日でやり切る**。これが力技。8時間集中投下するだけで、あなたは仕組みを持つ側に移る。

## ネタ4: 捨てる決断ができないと、仕組みは育たない

### 【哲学】

仕組みは「足し算」じゃない。**引き算**だ。

新しいツール、新しいフロー、新しい自動化を足していくと、複雑性が指数関数的に積み上がる。やがて誰も触れなくなる。これが世の中の「DX失敗」の正体。

**仕組みを育てる人は、定期的に捨てる人**。「もったいない」を捨てる。「過去の自分の選択」を捨てる。「他人が作った前提」を捨てる。

### 【俺の経験】

n8n。これは俺が2025年5月から12月まで**7ヶ月運用したワークフロー自動化ツール**。便利だった。動いてた。

でも、2025-12-29 に俺は1日で全部捨てた。

```
2025-12-29  a863917  refactor(mana): n8n完全廃止、GitHub直接コミットへ移行完了
2025-12-29  815c421  refactor(mana): n8n依存レガシーコードを削除
2025-12-29  ...     docs: n8n完全削除の完了レポートを追加
```

理由はシンプル。**「便利だから残す」じゃなくて「複雑性を生むから捨てる」**を選んだ。
- デバッグが地獄
- 環境変数管理が複雑
- スケールで詰まる

n8nを残してたら、今のMANA（PM工数75%削減）は絶対に無い。捨てたから次が来た。

X上にも似た声がある（@konkeisuke、17❤️）:
> 「AI時代は1周回って『属人化』が生存戦略になる。仕組み化できる業務ってAIができる業務と近いので自動化が進む」

→ 仕組み化できるものはAIに代替される。だから「仕組み×自分の判断」のセットでないと意味がない。**他人のツール（n8n）に乗ることをやめて、自分のコードで判断する**決断でもあった。

### 【あなたができること】

毎月最終金曜日を「**捨てる日**」にしてください。15分でいい。
1. 今月、何を始めたか書き出す
2. そのうち、来月も続けるべきものに○、迷うものに△、惰性のものに×
3. ×を全部消す。△は来月再判定する

これだけで、複雑性が指数関数的に増えなくなる。**捨てる訓練のない人は、仕組みを育てられない**。

## ネタ5: 属人性は仕組みの敵じゃない、形式知化する対象

### 【哲学】

「これは俺のセンスだから伝えられない」と言う人。  
「これはあの人にしかできない」と言われる人。

→ どっちも仕組みを持てない人の典型。

センス（属人性）は仕組みの**敵じゃない**。**形式知化する対象**。
- センスを「これはセンス」のまま持つ → スケールしない
- センスを「Markdownで書く」に変換する → AIに渡せる、引き継げる、仕組みに挿入できる

俺は2025/12/21の登壇でこう話した:

> 「立ち位置を定義する3要素：自分（Judgment Core）/ 仕事（Projects）/ 関係（People）」  
> 「AIの賢さ × **立ち位置整理** = 真の生産性向上」

立ち位置を整理できる人だけが、AIをパートナーにできる。**自分が誰かをAIに渡せる人**だけが、仕組みを持つ。

### 【俺の経験】

俺の workspace を見ると、こんなファイルがある。
- `IDENTITY.md` — 俺は何者か
- `CLAUDE.md` — AIに俺の判断基準を渡すファイル
- `_codex/` — 全プロジェクトの共有ナレッジ
- `decisions/` — 過去の意思決定ログ
- `people/` — 関係者一人ひとりのコンテキスト

これは「ドキュメント」じゃない。**俺の頭の中を、AIが読めるMarkdownに書き出したもの**。属人的な判断を、形式知に変換した結果。

だからAIに「これやって」って言うと、俺の判断基準で動く。**属人性を消したんじゃなくて、属人性を仕組みに挿入した**。

### 【あなたができること】

今週、`identity.md` を書いてください。10行でいい。
```
- 俺は誰か（職種じゃなく、価値観）
- 俺は何を大事にしてるか
- 俺は何を絶対やらないか
- 俺の判断基準トップ3
- 俺の今のフェーズ
```

書けたら、ChatGPT/Claudeに「これが俺の identity だ」と渡してから、何か質問してみてください。**回答が変わる**。これが「属人性の形式知化」の最小実証。

## ネタ6: 自動化しすぎるな — Tasteを挿入する場所を残す

### 【哲学】

仕組み化に夢中になる人ほど、**100%自動化を目指して死ぬ**。

俺は2026年4月のSNS自動化プロジェクトの会議でこう言った:
> 「自動化を進めすぎた結果、一旦お休みしようと思ってる。**ちょっと自動化しすぎるのも良くない**なって気づいた」

仕組み化の到達点は「全部AIがやる」じゃない。**「センスが要る部分だけ俺がやる、それ以外はAIがやる」の線引き**だ。

```
判断（Taste）→ 仕組みが実行 → 結果を判断（Taste）→ 仕組みを更新
```

仕組みの**最初と最後にTasteを挿入する**。間は仕組みが回す。これが正解。

### 【俺の経験】

X上にも警鐘の声がある（@ai_income_log、12❤️）:
> 「AI副業で初収益（197円）達成！でも、その裏側は盛大な『爆死』の記録でした…自動化に拘りすぎてスタンプ0枚、キャラが毎回別人になる地獄。仕組み化が好きな人ほど陥る『効率化の罠』」

俺もMANAで同じ罠にハマりかけた。「全部自動でPMする」を目指した時期があった。でも、**ストーリーの方向性とプロジェクトの優先度判断は俺がやる**ことに戻した。

今のMANA構造:
- **俺がやる（Taste挿入）**: 「何を作るか」「何を捨てるか」「誰と組むか」
- **MANAがやる（仕組み）**: タスク分解、進捗集約、議事録抽出、リマインド

この線引きを間違えると、「全部自動化したのに、出てくるアウトプットが全部ズレてる」ことになる。

### 【あなたができること】

3つだけ決めてください。「**これだけは絶対に俺がやる**」を3つ。

例:
- 採用の最終面接は俺がやる
- 顧客への謝罪は俺がやる
- 値段は俺が決める

書き出したら、それ以外は全部「AIに任せていい候補」。**自動化の線引きは、Taste領域の明示から始まる**。
