# なぜ仕様駆動では限界があるのか

<div class="two-col">
  <div class="panel">
    <h3>仕様駆動開発</h3>
    <ul>
      <li>仕様書をAIに渡す</li>
      <li>情報量が多い・圧縮されていない</li>
      <li>AIが全部を処理しなければならない</li>
      <li><strong>認知負荷が高い</strong></li>
      <li>意図が伝わらず、方向を間違える</li>
    </ul>
  </div>
  <div class="panel accent-panel">
    <h3>ストーリー駆動開発</h3>
    <ul>
      <li>物語（Story）をAIに渡す</li>
      <li>コンテキストが圧縮されている</li>
      <li>AIが「続きを想像」できる</li>
      <li><strong>認知負荷が低い</strong></li>
      <li>想像しやすいから、間違いにくい</li>
    </ul>
  </div>
</div>

<p class="note">本を半分まで読んで続きを想像できる。AIも同じ。</p>

---

# ストーリーで伝えると、AIが想像できる

<p class="big"><strong>「ストーリーで伝えると<br>想像しやすい。<br>だから間違いにくい。」</strong></p>

<p class="small">コンテキストの圧縮が本質。いかにエージェントにやりたいことを圧縮して伝えるか。<br>ストーリーがあると、AIが深掘りした時に「大元のストーリーからきっとこうだろう」と推測できる。</p>

---

# Frame / Story / Event — 圧縮の3層構造

<div class="codebox flow"><span class="accent">Frame</span>  ← 世界観・前提（ほぼ変わらない）
         どういう世界で、誰に、何を届けているか

<span class="accent">Story</span>  ← この世界でやること（horizon で粒度を分ける）
         northstar / annual / quarter / month / sprint

<span class="accent">Event</span>  ← 起きたことの記録（AIがオートで処理）
         ship / iteration / decision / フィードバック</div>

<p class="note">地図をAIに渡すと、AIがその全体感を理解しやすくなる</p>

---

# Storyからコードまで、レールがある

<div class="codebox flow">Story       ← 誰が・何を・なぜ（Howを書かない）
  ↓
Architecture ← 境界・責務（APIを書かない）
  ↓
Spec        ← API・スキーマ・フロー（初めてHowが確定）
  ↓
TDD / Code  ← Spec準拠のみ。ここで初めてAIを解き放つ</div>

<p class="note"><strong>フレームを飛ばしてストーリーを作らない<br>ストーリーを飛ばしてイベントを実行しない</strong></p>

---

# <span class="closing">AIに指示するな。<br>物語を渡せ。</span>

<p>Frame で世界観を固定する</p>
<p>Story でやることを圧縮する</p>
<p>Spec でレールを敷く</p>
<p>そこで初めて、AIが想像しながら正しく動く</p>
